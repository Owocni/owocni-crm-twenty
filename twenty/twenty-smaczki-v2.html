Kategoria I · 7 smaczków

Twenty 2.0 — co nowego

Release z kwietnia 2026 (Product Hunt #2 of the day, 349 upvotes). Pozycjonuje Twenty z "konfigurowalnego CRM-a" w "platformę programistyczną dla operacji biznesowych". Najmocniejsze rzeczy poniżej — wszystkie z opisu wersji.

2.0 AI · MCP 01

Native MCP Server w cloud workspace

Każdy workspace dostaje wbudowany Model Context Protocol server, do którego Claude / ChatGPT / Cursor podpinają się przez OAuth

To jest flagowy feature Twenty 2.0, a w poprzednim dokumencie został pominięty. Każdy cloud workspace Twenty ma natywny MCP server — to znaczy, że Twoje narzędzia AI (Claude Desktop, ChatGPT, Cursor) mogą bezpośrednio operować na Twoim CRM przez Model Context Protocol. Bez budowania własnej integracji, bez API keys w plain text, bez zewnętrznego brokera.

Podpięcie odbywa się przez OAuth z PKCE (smaczek 03), więc każdy AI agent dostaje token z uprawnieniami zalogowanego usera — nie globalny key. Schema CRM jest auto-discovered, więc Twoje custom fieldy są dostępne natychmiast po dodaniu, bez rekonfiguracji clienta MCP.

Społeczność dodatkowo wypuściła już własne implementacje MCP (m.in. twenty-crm-mcp-server i twenty-mcp z 29 toolami) dla self-host instancji, ale w cloud — natywne, zero-conf.

Jak użyć

Otwórz Claude Desktop → Settings → Connectors → dodaj Twenty workspace przez OAuth. Od tego momentu zapytanie typu "Pokaż mi wszystkie szanse powyżej 50k zł, których nie ruszałem od 2 tygodni" działa natywnie — Claude woła GraphQL, ty dostajesz wynik.

Dla zespołu sprzedaży to jest game-changer: zamiast uczyć ich filter buildera Twenty, uczysz ich pisać po polsku do Claude'a.

2.0 Performance 02

SDK split into subpaths — bundle 700× mniejszy

Zamiast ładować cały twenty-sdk, Logic Functions importują tylko subpath, którego potrzebują

Według developerów Twenty, ten jeden kawałek 2.0 zmniejszył bundle Logic Functions 700-krotnie w typowych przypadkach. To nie jest marketingowy okrągły numer — to fundamentalna zmiana dla serverless. Cold start Lambda przy bundle < 1MB vs > 700MB to różnica między 100ms a kilkoma sekundami.

Subpath imports w SDK:

  • twenty-sdk/define — defineX() functions (build-time)
  • twenty-sdk/cli — programmatic CLI (build, deploy, install)
  • twenty-client-sdk/core — typed GraphQL client dla data
  • twenty-client-sdk/metadata — workspace config + file uploads

Logic Function importujący tylko CoreApiClient nie ciągnie ze sobą całej kompilatorowej części SDK.

Jak użyć

Po prostu używaj subpath imports — to jest default w scaffolderze. Ale jeśli widzisz, że Twoja Logic Function ma duży bundle, sprawdź, czy nie importujesz przez root 'twenty-sdk'. Tree-shaking nie zawsze złapie wszystko, więc jawne subpath jest bezpieczniejsze.

2.0 Bezpieczeństwo 03

OAuth 2.0 z PKCE i RFC 7591

Pełen industry-standard OAuth flow: Authorization Code z PKCE, Client Credentials, dynamic client registration

Twenty 2.0 ma cztery rzeczy, które razem tworzą enterprise-grade auth:

  • Authorization Code Flow z PKCE — dla user-facing aplikacji. Token wystawiany on-behalf-of usera, refreshable.
  • Client Credentials Flow — dla server-to-server bez kontekstu usera. Token z workspace-level access.
  • Dynamic Client Registration (RFC 7591) — programmatically rejestrujesz client przez POST /oauth/register, zero ręcznej konfiguracji w panelu.
  • Server Discovery — GET /.well-known/oauth-authorization-server zwraca metadata, endpoints, supported grants. Pozwala generic OAuth clientom adaptować się automatycznie.

PKCE jest obowiązkowy dla public clients od 2.0 — to jest standardowa zmiana, którą Microsoft i Google wymuszały wcześniej. Twenty to zaadoptowało zgodnie z RFC 9728.

Dwa scope: api (full read/write) i profile (user info).

Jak użyć

Dla każdej zewnętrznej aplikacji, która ma działać "on-behalf-of" usera (np. Slack bot, mobilna apka, zewnętrzny dashboard) — Authorization Code z PKCE.

Dla server-to-server (Sortownia → Twenty) — Client Credentials, ale rozważ też API keys jeśli sprawa jest prosta.

OAuth jest też mechanizmem za MCP Server (smaczek 01) — gdy podpinasz Claude Desktop, to pod spodem dzieje się Authorization Code Flow.

2.0 DX 04

npx create-twenty-app — scaffolder w 30 sekund

Jedno polecenie tworzy gotowy projekt z SDK, Vitest, GitHub Actions CI, scaffoldowaną rolą

Inspirowane create-react-app, create-next-app. Komenda npx create-twenty-app@latest my-app generuje:

  • Strukturę folderów (src/objects/, src/fields/, src/logic-functions/, src/front-components/...)
  • application-config.ts z defineApplication() + UUID
  • Default role w src/roles/default-role.ts
  • Vitest configurację w vitest.config.ts
  • .github/workflows/ci.yml z spawn-twenty-docker-image action
  • tsconfig.json, package.json, .gitignore

Po cd my-app && yarn twenty dev masz live sync z lokalnym Twenty server'em. Standard 5-minute-quickstart bez frustracji.

Jak użyć

Każda nowa aplikacja w Twoim ekosystemie startuje od npx create-twenty-app. Nie pisz boilerplate od zera, nie skopiuj poprzedniej aplikacji — zawsze świeży scaffold.

Dla zespołu o różnych poziomach zaawansowania: scaffolder eliminuje całą warstwę "który config jest poprawny", "jak wygląda canonical struktura folderów".

2.0 Enterprise 05

Self-host licensing & billing

Wbudowane procesy: checkout, activation, status queries, seat management — dla self-hosted instancji

2.0 wprowadziło commercial functionality dla self-hosting'u. Przed 2.0 — self-host był 100% darmowy (open source), bez żadnej warstwy komercyjnej. Teraz Twenty oferuje:

  • Checkout — wykupienie licencji z poziomu instancji
  • Activation — aktywacja licensed features (np. SSO, advanced permissions)
  • Status queries — sprawdzanie aktualnego stanu subskrypcji
  • Seat management — zarządzanie liczbą userów

Cloud Pro jest $9/user/month (yearly), Organization tier $19/user/month z SSO.

To otwiera ścieżkę dla firm, które chcą self-host (compliance, data residency) ale potrzebują też enterprise features. Wcześniej musiałeś wybrać: cloud z featurami i lock-in, lub self-host bez featurów.

Jak użyć

Dla firm z compliance requirements (sektor publiczny, finanse, ochrona zdrowia) — self-host + Organization licensing. Zachowujesz dane na własnej infrastrukturze, dostajesz SAML/OIDC SSO i support.

Dla agencji takiej jak Owocni — cloud Pro $9/user jest wystarczający. Self-host warto rozważyć dopiero przy 20+ użytkownikach i specyficznych wymaganiach.

2.0 DevOps 06

AWS EKS targets + ESM module cache

Nowe Docker targets dla Kubernetes deployment, plus cache modułów ESM między wywołaniami Lambda

Dwie zmiany infrastrukturalne, które razem znaczą "Twenty jest gotowe na production scale":

  • AWS EKS targets — gotowe Docker images z konfiguracją zoptymalizowaną pod Kubernetes na AWS. Helm charts i deployment manifesty out-of-the-box.
  • ESM module cache w Lambdzie — Twenty serverless workers (gdzie biegają Logic Functions) cachują skompilowane moduły między wywołaniami. To redukuje koszt cold startu i poprawia steady-state performance.

Dodatkowo 2.0 fixuje konkretne database query inefficiencies (Cartesian products powodujące timeouty).

Jak użyć

Dla self-host'u Twenty na produkcji w skali (powyżej kilku set userów) — EKS targets są domyślną ścieżką. Dla mniejszych instancji Docker Compose nadal wystarcza.

Cache ESM jest transparentny — nie musisz nic konfigurować. Po prostu Twoje Logic Functions startują szybciej.

UX 07

Cmd+K Command Menu — uniwersalny entry point

Klawiatura zamiast myszy. Custom commands z aplikacji. Manual workflow triggers

To jest kluczowy element UX Twenty, który w poprzednim dokumencie ledwo zaznaczyłem. Cmd+K (lub Ctrl+K na Windows) otwiera command menu, które jest centralnym entry pointem do całego workspace'u:

  • Navigation — przeskocz do dowolnego rekordu wpisując fragment nazwy
  • Create record — szybkie tworzenie firmy/osoby/szansy bez klikania w UI
  • Run workflow — Manual Trigger workflows pojawiają się w menu, można je odpalić jednym Enter'em
  • Custom commands — Twoje aplikacje (Front Components) mogą rejestrować custom items w menu
  • Action shortcuts — typowe operacje jak "go to settings", "create view", "import CSV"

Dla power userów to jest 3-5× przyspieszenie codziennej pracy. Notion, Linear, Raycast — wszyscy zrobili tak samo, bo to działa. Twenty robi to świadomie ("Notion for CRM").

Jak użyć

Trenuj zespół sprzedaży na Cmd+K od pierwszego dnia. To jest największy single-handed boost produktywności w Twenty.

Dla custom workflows: Manual Trigger z trybem Global + odpowiednia ikona = zespół ma jednoklawiszową automatyzację dla typowych operacji ("dodaj lead z Sortowni", "wyślij case study", "zarezerwuj kalendarz").

Kategoria II · 5 smaczków

AI & MCP

Twenty pozycjonuje się jako platforma AI-native. Wszystko AI-related jest first-class citizen w architekturze: agenty żyją w manifeście, prompty są kodem, MCP server jest gateway'em. Permission model dotyczy też agentów — bez wyjątków.

AI 08

AI Chatbot z page context

"Summarize my interactions with this person" — chatbot wie, na której karcie jesteś

Inteligentny asystent, który pozwala pytać o dane CRM po polsku/angielsku zamiast budować filtrów. Co ważne — ma page context: gdy jesteś na karcie firmy, "this company" referuje do niej. Gdy jesteś na opportunity, "this deal" jest odpowiednio kontekstualizowane.

Capabilities:

  • Natural language queries — "Pokaż mi szanse powyżej 50k zł", "Znajdź kontakty bez maila z ostatnich 2 tygodni"
  • Insights — "Jaki jest mój total pipeline?", "Ile dealów zamknęło się w zeszłym miesiącu?"
  • Page context — "Podsumuj moje interakcje z tym kontaktem"
  • Conversational — kolejne pytania zachowują kontekst poprzednich

Status: w developmencie, beta wkrótce. Działa w obrębie permission modelu (smaczek 11), więc nie pokaże danych, których user nie powinien widzieć.

Jak użyć

Gdy będzie dostępne, użyj go jako podstawowego entry pointu dla zespołu sprzedaży. Zamiast trenować ich filter buildera, niech piszą zapytania. Page context znaczy, że gdy patrzą na konkretnego leada, mogą zapytać "kiedy ostatnio go ruszaliśmy?" bez selektowania ID.

AI 09

Skills & Agents jako entity workspace'u

defineSkill (instrukcje) + defineAgent (gotowy asystent) jako TypeScript files w git

Skill to reusable zestaw instrukcji ("jak prowadzić sales outreach"), Agent to gotowy asystent z własnym system promptem i opcjonalnym modelId. Oba dostają universalIdentifier, label, ikonę, description.

Status: alpha, ale działa. Workspace może mieć wielu agentów obok siebie, każdy z własnym scope'em.

// src/agents/lead-qualifier.ts
import { defineAgent } from 'twenty-sdk/define';

export default defineAgent({
  universalIdentifier: '...',
  name: 'lead-qualifier',
  label: 'Lead Qualifier',
  prompt: `Jesteś specjalistą od kwalifikacji leadów dla agencji web premium...`,
  modelId: 'claude-sonnet-4',
});
Jak użyć

Wszystkie prompty Twojego CRM-u w src/skills/ i src/agents/. Iteracje promptu = commity w git. Deploy promptu = deploy aplikacji. Cofnięcie promptu = rewert commita. Eliminuje całą warstwę zewnętrznych narzędzi do prompt management (Mem0, Pinecone, własne JSON-y).

AI 10

isTool: true — Logic Function staje się tool dla AI

Jeden flag i serverowy TypeScript jest discoverable przez agentów Twenty

Definiujesz funkcję serverową (np. "wzbogać firmę o dane z BIK/NIP"), dajesz jej isTool: true, i staje się dostępna dla każdego AI agenta w workspace'ie. Twenty automatycznie generuje toolInputSchema ze statycznej analizy kodu.

Funkcja może jednocześnie mieć trigger HTTP/cron/database event i być tool'em. Jedna implementacja, dwa entry pointy.

export default defineLogicFunction({
  universalIdentifier: '...',
  name: 'enrich-company',
  description: 'Wzbogaca firmę o dane z BIK, NIP, KRS',
  isTool: true,
  handler,
  httpRouteTriggerSettings: { path: '/enrich', httpMethod: 'POST' },
});
Jak użyć

Dobry description jest krytyczny — z dokumentacji: "AI agents rely on the function's description field to decide when to use the tool. Be specific about what the tool does and when it should be called."

Każda Twoja integracja (lookup w BIK, scrape strony, generuj wycenę PDF) — Logic Function z isTool: true. AI Chatbot Twenty sam jej użyje gdy kontekst tego wymaga.

Bezpieczeństwo AI 11

AI w ramach permission modelu

Agent widzi tylko obiekty i pola, do których ma uprawnienia. Sensitive data chronione even when AI is involved

To jest filozoficzna decyzja Twenty, która ma poważne implikacje. Z dokumentacji: "AI in Twenty respects your permission model. Agents can only access objects and fields that the user (or role) has permission to view."

Co to znaczy w praktyce:

  • Jeśli sales rep nie widzi pola "annual revenue" firmy — AI agent działając on-behalf-of niego też nie widzi.
  • Jeśli aplikacja ma rolę z dostępem tylko do Companies — agent w niej zarejestrowany nie widzi People.
  • Jeśli dla custom field jest ograniczenie field-level — AI nie pokaże go w summary.

To jest fundamentalnie inne podejście niż "AI ma admin access". Tu permission to walls dotyczą nawet bot'ów.

Jak użyć

Każda aplikacja, która ma agentów, dostaje swoją własną rolę przez defineRole(). Daj tej roli minimalny zakres: tylko obiekty i pola, których agent rzeczywiście potrzebuje do swojej pracy.

Jeśli budujesz agenta "Lead Qualifier" — daj jej dostęp do Opportunities i Companies, ale nie do Tasks ani Notes z prywatnych spotkań. Ograniczenie scope'u = ograniczenie blast radius.

AI · Workflows 12

Auto-Reply via AI Agent + In-Reply-To threading

AI triażuje email (skip / reply), drafuje odpowiedź, Send Email z headerMessageId daje threading w Gmailu

Twenty publikuje konkretny step-by-step recipe, w którym pokazuje jak zbudować AI auto-reply na inbound emails:

  1. Trigger: Record is Created → Object: Messages
  2. Search Records na Message Participants z Role = "From" → bierze sender's email i nazwę
  3. AI Agent action z promptem typu: "Read this email. If spam/newsletter, respond SKIP. Otherwise write 3-4 sentence reply."
  4. If/Else branch: response does not contain SKIP
  5. Send Email with In-Reply-To: {{trigger.headerMessageId}} + Subject: Re: {{trigger.subject}}

Haczyk: Gmail wymaga, żeby subject zaczynał się od "Re:" żeby pokazać reply w wątku. Bez tego In-Reply-To header nie wystarczy.

Jak użyć

Każdy automated reply z Twenty (auto-confirm, AI-drafted response, routing notification) — zawsze przez headerMessageId threading. Bez tego automaty wyglądają jak spam — nowy thread za każdym razem.

Sender pobierasz przez Search Records na Message Participants (Role = From), nie z samego Message. To jest częsty pułapka.

defineEntity + AST detection

Architektura 14

Jak użyć

Bezpieczeństwo 17

Lifecycle 19

Pre-install & post-install hooki

Pre-install może abortować upgrade, post-install seeduje dane. Async z retry albo sync

  • Pre-install — uruchamiany przed migracją metadanych, na starym schemacie. Synchroniczny i blokujący. Jeśli rzuci wyjątek, instalacja jest abortowana, workspace zostaje nietknięty. Idealne do backupu danych, archiwizacji, walidacji kompatybilności.
  • Post-install — uruchamiany po migracji, na nowym schemacie. Domyślnie asynchroniczny (na message queue z 3 retry), opcjonalnie synchroniczny. Używany do seedingu danych, rejestracji webhooków zewnętrznych, konfiguracji workspace'u.

Oba hooki dostają InstallPayload z { previousVersion?, newVersion } — możesz odróżnić fresh install od upgrade.

Jak użyć

Pre-install dla backupu pola, które usuwasz w v2: skopiuj notes do description zanim migracja zrzuci kolumnę.

Post-install dla seedingu domyślnych etapów pipeline'u, default views, demo content.

Workflows 23

Iterator action — pętla po array

Search Records → Iterator → bulk operations na każdym znalezionym rekordzie

Iterator bierze array (z Search Records, webhook payload, Code action output) i wykonuje wszystkie kolejne actions per item. Wewnątrz pętli dostęp do {{iterator.currentItem}} i {{iterator.index}}.

1. Search Records (People, Company = "Acme")
2. Filter ({{searchRecords.length}} > 0)
3. Iterator ({{searchRecords}})
   └── Send Email
       To: {{iterator.currentItem.email}}
       Subject: Hello {{iterator.currentItem.firstName}}!

Pułapka: zawsze sprawdzaj length > 0 przed Iteratorem, bo na empty array Iterator wyrzuca błąd.

Search Records ma soft limit 200 records. Dla większych operacji — batche.

Jak użyć

Bulk update overdue tasks: Search Records (Tasks, Due < Today, Status ≠ Completed) → Iterator → Update Record (Status = Late).

Webhook order with multiple items: Webhook → Iterator on body.items → Create Record per item.

Conditional inside loop: Iterator → Filter (currentItem.email is not empty) → Send Email.

Workflows 24

Code Actions — custom JavaScript w workflow

Inline JavaScript dla obliczeń, transformacji, custom logiki — bez build pipeline'u, bez deploya

Workflow editor pozwala dodać Code Action, gdzie piszesz funkcję JavaScript inline:

export const main = async (params: {
  lastEnteredPreviousStage: Date;
}): Promise<object> => {
  const { lastEnteredPreviousStage } = params;
  const now = new Date();
  const diffTime = Math.abs(now.getTime() - new Date(lastEnteredPreviousStage).getTime());
  return {
    now: now.toISOString(),
    daysInPreviousStage: Math.ceil(diffTime / (1000 * 60 * 60 * 24)),
  };
};

Output zwracany z funkcji jest dostępny w kolejnych krokach workflow przez {{Code action.now}}, {{Code action.daysInPreviousStage}}.

Różnica vs Logic Function: Code Action to inline w workflow, bez własnego deploy cycle. Dla prostych transformacji to wygodniejsze. Dla powtarzalnych operacji w wielu workflow — Logic Function.

Jak użyć

Pattern Track time-in-stage (smaczek 30) używa Code Action dla obliczania now - lastEnteredPreviousStage.

Custom field formatting, conditional logic, math — Code Action.

External API call, complex business logic — Logic Function.

Workflows 25

Workflow Branches z If/Else

Conditional logic, parallel paths, nested branches w jednym workflow

If/Else action w workflow editorze pozwala rozgałęzić logikę. Każda gałąź ma swoje filter conditions i actions. Branches mogą być nested.

Use case'y:

  • Routing per industry: lead z healthcare → Sales rep A, lead z fintech → Sales rep B
  • Conditional notifications: high-value deal > 100k zł → notify CEO, lower → standard
  • AI triage decisions: AI agent zwraca SKIP lub reply text → If/Else na presence "SKIP"
  • Multi-stage pipeline: jeden workflow z gałęzią per stage (track time-in-stage pattern)

Branches nie muszą się łączyć z powrotem. Każda action po If/Else może być końcem gałęzi.

Jak użyć

Skomplikowane workflow z wieloma punktami decyzyjnymi — jeden workflow z branches, nie wiele oddzielnych. Łatwiej utrzymać, łatwiej audytować, łatwiej zmienić.

Bezpieczeństwo 32

Webhooks z HMAC SHA256

Outgoing notifications na każde create/update/delete, podpisane jak Stripe

Każdy outgoing webhook ma dwa headery: X-Twenty-Webhook-Signature (HMAC SHA256) i X-Twenty-Webhook-Timestamp. Walidacja: tworzysz string {timestamp}:{payload}, liczysz HMAC z secretem, porównujesz signature.

Eventy: wszystkie object types (włącznie z custom), trzy operacje: created, updated, deleted.

const stringToSign = `${timestamp}:${JSON.stringify(req.body)}`;
const expected = crypto.createHmac('sha256', secret)
  .update(stringToSign).digest('hex');
const valid = crypto.timingSafeEqual(
  Buffer.from(expected, 'hex'),
  Buffer.from(req.headers['x-twenty-webhook-signature'], 'hex')
);
Jak użyć

Dla external systemów reagujących na zmiany w Twenty (notification service, data warehouse, marketing automation) — webhook z signature verification. Dla logiki przed wysłaniem — Workflow + HTTP Request action.

Migration 34

CSV import z Upsert i field mapping

Match-or-create, duplicate detection, error handling w UI przed commitem

Twenty ma flow CSV importu z:

  • Field mapping — mapowanie kolumn CSV na pola obiektu, w UI
  • Upsert operation — update istniejący record jeśli match, create jeśli nie. Klucz match'owania konfigurowalny.
  • Duplicate detection — Twenty znajduje potencjalne duplikaty przed commitem
  • Uniqueness constraints — możesz wymusić unique values per field
  • Error handling w UI — review and fix import errors przed kliknięciem confirm
  • Import relations — linkowanie records między obiektami via CSV

Dla większych migracji — API import handler z 50,000+ records.

Jak użyć

Migracja z Salesforce / HubSpot zaczyna się od export do CSV → field mapping w Twenty → review duplicates → commit. Dla 50k+ records — API handler.

Update bulk: ten sam mechanizm. Zamiast pisać workflow na update, exportujesz CSV, modyfikujesz, importujesz z Upsert — prościej dla one-off updates.

Kategoria VII · 3 smaczki

Docs & LLM-friendliness

Najmocniejszy wyróżnik Twenty dla zespołów AI-driven. Nie dlatego że Twenty wbudowało AI w produkt (każdy to robi), ale dlatego że infrastruktura pod LLM-y jest celowa: docs, constraints, CLI — wszystko zaprojektowane pod use case "model pisze kod pod Twenty".

LLM · DevEx 35

llms.txt — kompletny index docs dla LLM-ów

Każda strona ma swój markdown odpowiednik .md, jeden plik indeksuje wszystko

Pod adresem docs.twenty.com/llms.txt — kompletny indeks całej dokumentacji w czystym markdown. Każda strona docs ma swój ekwiwalent .md: /getting-started/core-concepts/ai.md, /developers/extend/apps/front-components.md itd.

To jest rzadka rzecz — większość firm tego nie robi. LLM dostaje surowe materiały bez parsowania DOM-u i bez halucynacji o przestarzałych URL-ach.

Jak użyć

Wpinaj llms.txt jako stały kontekst do system prompta każdego agenta, który ma generować kod pod Twenty. Konkretne pliki, które warto trzymać domyślnie wpięte: style-guide.md, front-components.md, data-model.md, logic-functions.md.

LLM constraint 36

Style Guide jako twarde constraints dla LLM-a

Zero raw values, only theme tokens, only twenty-sdk/ui imports — enforced przez oxlint

Najważniejsze reguły z docs.twenty.com/developers/contribute/style-guide.md:

  • Zero raw values w styled components — żadnych padding: 13px, color: '#3a4f8b'. Wszystko przez theme.spacing(n), theme.color.*.
  • Linaria styled z prefixem Styled
  • Functional components z export function, nigdy export default const
  • Type, nie interface; string literals, nie enums
  • Brak React.FC, brak prop spreadingu
  • Komponenty wyłącznie z twenty-sdk/ui lub @/ui/*
  • oxlint wymusza większość tego automatycznie
Jak użyć

Te reguły są idealne jako constraints w system prompt LLM-a. Model nie może wymyślić padding: 13px, bo wie, że to violation. Pętla weryfikacyjna w Twojej Konstytucji v2.9 pasuje wprost — gates G1–G4: regex na px|rem|#[0-9a-f], imports z twenty-sdk/ui, defineFrontComponent wrapper, oxlint.

Kategoria VIII · 2 smaczki

Operations & Enterprise

Sprzedaż firmom z compliance requirements. Te rzeczy są mniej "wow!", ale bez nich Twenty nie wchodzi do enterprise — i bez nich też ciężko dla zespołu w Polsce udowodnić klientowi z sektora finansów, że nie potrzebują Salesforce.

Enterprise 38

SSO SAML/OIDC + audit logs + field-level permissions

Enterprise auth, kto co zrobił, granularne uprawnienia

Trzy filary enterprise readiness Twenty:

  • SSO — SAML i OIDC dla single sign-on. Konfiguracja w Settings → Permissions & Access.
  • Audit logs — kto co zrobił, kiedy. Każda zmiana rekordu, każda akcja administracyjna.
  • Role-based access control na trzech poziomach: objects, fields, individual records. Field-level permissions znaczy, że możesz ukryć "salary" przed wszystkimi poza HR.

SSO i niektóre advanced permissions są w Organization tier ($19/user/month) lub przez self-host activation.

Jak użyć

Dla każdego klienta z sektora finansów / ochrony zdrowia / publiczne — SSO + audit logs są must-have, nie nice-to-have. Nie podlega negocjacji.

Field-level permissions: ukryj wrażliwe pola (salary, NDA terms, SSN) przed userami, którzy ich nie potrzebują. Default'em widzą wszyscy, więc trzeba aktywnie ograniczać.

DevOps 39

Vitest + spawn-twenty-docker GitHub Action

Ephemeral Twenty w CI runnerze, programmatic appBuild/Deploy/Install API

Scaffolder Twenty generuje od razu .github/workflows/ci.yml, który używa twentyhq/twenty/.github/actions/spawn-twenty-docker-image@main. Akcja w runnerze:

  1. Spawn-uje świeżą instancję Twenty w Dockerze
  2. Output-uje server-url i access-token
  3. Twoje testy Vitest dostają te wartości jako env vars

SDK eksportuje programmatic API: appBuild(), appDeploy(), appInstall(), appUninstall(). Możesz w jednym test file zrobić build → deploy → install → assert → uninstall na realnym Twenty.

Żadnych secrets do konfiguracji w GitHub. Pojedyncza ephemeral instancja per build. Pinuj wersję Twenty przez TWENTY_VERSION: 2.0.0.

Jak użyć

Każda Twoja aplikacja ma integration testy weryfikujące że post-install seeduje dane, że HTTP triggers przyjmują payload, że Front Component renderuje się poprawnie. Bez ręcznej weryfikacji na produkcji.

Kategoria IX · 2 smaczki

Kultura & filozofia

Na koniec — to, czego nie znajdziesz w docs, ale co determinuje całą resztę. Dlaczego Twenty wygląda tak jak wygląda. Dlaczego ludzie wybierają go nad Salesforce, nawet wiedząc, że feature parity jeszcze nie ma.

Kultura 40

#1 open-source CRM na GitHubie + zero vendor lock-in

MIT-style license, self-host w Docker Compose, eksport bez ograniczeń

Twenty pozycjonuje się explicit przeciwko Salesforce / HubSpot. Z ich strony:

  • Open source — pełen kod na GitHubie, każdy może forkować
  • Self-hostable — single Docker Compose command i działa na Twojej infrastrukturze
  • No proprietary tooling — TypeScript, React, PostgreSQL. Stack, który zna każdy dev.
  • Export anytime — Twoje dane są zawsze Twoje, w standardowych formatach
  • Avoid lock-in — jeśli kiedykolwiek będziesz chciał odejść, masz code i dane. Worst case — fork i hostuj samemu.

Case studies z ich strony: AC&T (zastąpili shuttered vendor CRM, redukcja kosztów >90%), W3villa (zbudowali "W3Grads" — AI mock interviews jako Twenty platform), NetZero (modular setup dla carbon credits + agricultural products).

Dla developerów to jest argument fundamentalny. "Build on top of it" ma sens tylko jeśli platforma nie zniknie z twojej road mapy z dnia na dzień.

Jak użyć

To jest argument sprzedażowy dla klienta: "Twoje dane są Twoje, kod jest open, możesz hostować sam, możesz forkować." Vs Salesforce: "Płacisz $300/user/month, dane są w ich chmurze, kod jest closed, jeśli odejdziesz — zostaje wybić zęby na CSV exporcie i czternastu integracjach do przepisania."

Dla agencji premium sprzedającej swoim klientom CRM-y — Twenty pozwala oferować coś, czego konkurencja nie oferuje. Własność.

Kultura 41

"Notion for CRM" — design philosophy

Clean UI, keyboard-first, real-time updates, bez "feature creep" sprzed dekady

Twenty świadomie modeluje UX po Notion / Linear / Raycast. Konkretne implikacje:

  • Cmd+K jako uniwersalny entry point (smaczek 07)
  • Inline editing bez "edit mode → save" — klikasz w pole, wpisujesz, zapisuje się
  • Real-time updates — zmiany pojawiają się natychmiast u innych userów
  • Minimalna nawigacja — sidebar, content, command menu. Bez 47 zakładek.
  • Customowe ikony i kolory per item w Favorites — workspace ma osobowość
  • Keyboard-first — wszystko ma shortcut, mysz jest opcjonalna

Praktyczna konsekwencja: GTM teams adopcja jest wyższa. Zespół sprzedaży w Salesforce narzeka. Zespół w Twenty pyta o nowe features. To jest realna różnica.

Twenty 2.0 to wzmocniło — Cmd+K teraz może być customizowane przez aplikacje (widget komendy z Front Components), navbar może mieć custom buttons, page layouts są w pełni configurable.

Jak użyć

Wdrażając Twenty w zespole Owocni — naucz ich Cmd+K pierwszego dnia. Pin views w sidebarze. Dodaj custom buttons w navbar dla typowych akcji. Zespół pokocha to narzędzie zamiast je tolerować.

Dla klientów premium — to jest argument różnicujący. "Twój zespół faktycznie zacznie używać CRM-a, nie tylko będzie się go bał".

✓ Kluczowe odkrycie: „Bitrix" to Supabase

Moduł leadów (/lead), którego używacie na co dzień, jest w 100% oparty na Supabase. Bitrix24 API nie jest wywoływany z żadnego elementu tego modułu — ani z kanbana, ani z formularzy, ani z kontekstowego menu, ani z dialogów wygrania/przegrania leada.

Nazwa „better-bitrix" jest historyczna. Aplikacja była budowana jako nakładka na Bitrix24, ale z czasem praktycznie cały pipeline leadowy został przeniesiony na Supabase. Stare endpointy Bitrix24 (api/newLead/v1-v3) nadal istnieją w kodzie, ale nikt ich nie wywołuje z frontendu.

Gdzie jeszcze żyje Bitrix24 API?

Endpointy api/newLead/, api/newLead/v2/, api/newLead/v3/ nadal importują @2bad/bitrix i wywołują crm.lead.get, crm.company.add, bitrix.leads.update. Jednak żaden z nich nie jest wywoływany z frontendu /lead ani z nowego flow tworzenia leadów (createLeadv2.ts). To martwy kod — relikt z czasów, gdy leady szły przez Bitrix.

⚠ Ważne: różnica terminologiczna „Deal" vs „Lead"

EVENT_CONTRACT_OWOCNI.md używa słowa „Deal" — ale to nomenklatura Twenty, nie Bitrix24.

W Twenty CRM główny obiekt pipeline'u sprzedażowego nazywa się Opportunity (alias „Deal"). To jest odpowiednik tego, co w better-bitrix nazywacie „leadem" — wpis w kanbanie ze statusami: nowy → kwalifikowany → wygrany/przegrany.

Wasze „deale" w Bitrix24 to zupełnie inny koncept — workflow realizacji projektu po wygranym leadzie. Dokumentacja Twenty tego w ogóle nie porusza. To osobny system, osobny temat.

Sortownia (sGTM) + Robot (GCF) — rola w ekosystemie

KomponentCo robiCzy tworzy leada w CRM?
Sortownia (Stape sGTM Custom Template) Przyjmuje generate_lead z formularzy owocni.pl. Normalizuje PII (email, phone). Tworzy profil w identity_map (Stape Store). Dodaje tasks do task_queue. NIE — zapisuje do Stape Store, nie do CRM.
Robot (Google Cloud Function) Przetwarza task_queue z Sortowni. Wysyła eventy do GA4 Measurement Protocol, Meta CAPI, Google Ads Enhanced Conversions. NIE — wysyła do platform marketingowych.

✂ Co NIE trzeba migrować

Twenty ma natywnie

  • Kanban / Table / Calendar views — gotowe, z filtrami, sortowaniem, aggregacjami. Nie trzeba budować.
  • Cmd+K Command Menu — zastąpi custom search i nawigację.
  • REST + GraphQL API — auto-generowane. Nie trzeba pisać endpointów.
  • Webhooks z HMAC — gotowe. Nie trzeba budować infrastruktury powiadomień.
  • Real-time — natywny WebSocket. Nie trzeba konfigurować Supabase Realtime.
  • MCP Server — AI agenty mogą operować na CRM natywnie.
  • Email & Calendar Sync — Google / M365 out-of-the-box (przydatne przy leadach, nie trzeba custom IMAP).

Poza scope'em / do porzucenia

  • Bitrix24 (deale + faktury) — osobny system, osobna migracja w przyszłości.
  • /lead2 — nieużywany moduł w better-bitrix.
  • api/newLead/v1-v3 — martwy kod Bitrix w better-bitrix.
  • api/bitrix/* — stare proxy do Bitrix24.
  • lib/bitrix/ — cały connector Bitrix.
  • queries/lead_old/ — stare query hooki.
  • Inne moduły better-bitrix (inbox, chat, WordPress, tools) — nie w scope'ie, mogą być przeniesione później.

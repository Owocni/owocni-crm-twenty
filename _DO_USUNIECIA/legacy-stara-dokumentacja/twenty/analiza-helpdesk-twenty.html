Helpdesk w Twenty CRM

Ocena możliwości przeniesienia funkcjonalności helpdesku z better-bitrix na Twenty

Przygotowano: 25 maja 2026

📋 Kontekst

Rozważamy migrację z autorskiego CRM (better-bitrix, opartego na Next.js + Supabase) na Twenty CRM — open-source'owy, AI-natywny CRM pozycjonowany jako platforma deweloperska.

Jednym z kluczowych modułów w better-bitrix jest helpdesk — wewnętrzny system ticketowy do obsługi zgłoszeń klientów. Niniejszy dokument analizuje, czy Twenty oferuje taką funkcjonalność i jakie są opcje jej odtworzenia.

⚠ Kluczowy wniosek

Twenty nie posiada wbudowanej funkcjonalności helpdesku.
W dokumentacji (EVENT_CONTRACT_OWOCNI.md, twenty-smaczki-v2.html) nie znaleziono żadnej wzmianki o: helpdesk, ticket, zgłoszenie, customer support, customer service.
Ale można ją zbudować.
Twenty 2.0 oferuje rozbudowaną platformę deweloperską (custom objects, workflows, logic functions, front components), która pozwala odtworzyć helpdesk jako rozszerzenie.

🔧 Narzędzia Twenty do budowy helpdesku

Twenty 2.0 (kwiecień 2026) oferuje następujące mechanizmy, które można wykorzystać:

MechanizmCo daje
Custom Objects
defineObject()		 Tworzenie obiektu "Ticket" z dowolnymi polami — automatycznie dostaje REST/GraphQL API, widoki tabelaryczne i Kanban
Workflows
(no-code)		 7 typów triggerów (Created, Updated, Scheduled, Webhook…), akcje warunkowe, Code Actions — idealne do automatyzacji statusów i powiadomień
Logic Functions
(serverless TS)		 Triggery HTTP, cron i database event — do integracji z pocztą, webhooków, custom logiki
Front Components
(sandboxed React)		 Widgety dashboardowe, zakładki na stronie obiektu, panele boczne — do custom UI helpdesku
REST + GraphQL API Auto-generowane dla każdego obiektu (w tym custom) + OpenAPI spec
Webhooks + HMAC Do integracji z zewnętrznymi serwisami (mail server, Slack, itp.)
MCP Server AI-agenty mogą operować na ticketach przez Claude/ChatGPT/Cursor
Permissions Uprawnienia na poziomie pól (plan Enterprise)

💡 Możliwe podejścia

Opcja A: Helpdesk jako rozszerzenie Twenty

Zbudowanie pełnego helpdesku na platformie Twenty za pomocą custom objects, workflows, logic functions i front components.

+ Wszystko w jednym systemie + Wspólna baza kontaktów i firm + AI-agenty mają dostęp do ticketów
− Wymaga znacznego nakładu pracy deweloperskiej − Integracja z pocztą będzie najtrudniejsza

Opcja B: Twenty CRM + zewnętrzny helpdesk

Twenty jako CRM sprzedażowy, a helpdesk na dedykowanym narzędziu open-source (np. Zammad, FreeScout, Trudesk) zintegrowanym przez API/webhooks.

+ Gotowe rozwiązanie helpdesk out-of-the-box + Szybsze wdrożenie + Specjalizowane narzędzia do supportu
− Dwa osobne systemy do utrzymania − Wymaga integracji (synchronizacja kontaktów, linki)

Opcja C: Zachowanie better-bitrix tylko jako helpdesk

Migracja CRM-u do Twenty, ale utrzymanie modułu helpdesku w better-bitrix (okrojona wersja aplikacji).

+ Zero pracy nad nowym helpdeskiem + Sprawdzone rozwiązanie
− Utrzymanie dwóch aplikacji − Brak wspólnego kontekstu klienta − Dług technologiczny

📊 Podsumowanie

Rekomendacja: Decyzja zależy od priorytetów zespołu. Jeśli zależy nam na spójnym ekosystemie i mamy zasoby deweloperskie — Opcja A (helpdesk na Twenty) daje najlepszą długoterminową wartość. Jeśli potrzebujemy szybkiego wdrożenia — Opcja B (zewnętrzny helpdesk) jest pragmatycznym kompromisem.
